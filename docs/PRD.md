## 1. 프로젝트 개요

### 한 줄 요약

세탁기에 붙인 QR을 스캔하면 → 누구의 세탁물인지 + 바구니가 뭔지 바로 확인할 수 있는 웹 서비스

### 문제

- 공용 세탁실에서 세탁 끝난 세탁물의 주인을 알 수 없음
- 누구 바구니인지 몰라서 세탁기 위나 바닥에 방치 → 오염 발생

### 해결

- 세탁기마다 QR 스티커 부착
- QR 스캔 → 닉네임 + 바구니 사진 등록/확인
- 다른 사람이 새로 등록하면 이전 사진 자동 삭제 (용량 최소화)

### 사용자

종근당 학사 거주자 (검증된 구성원이므로 로그인 불필요)

### 핵심 설계 원칙

- **관리자 없음** — 로그인, 관리자 페이지, 권한 관리 일절 없음
- **DB 사전 등록 없음** — QR URL만 있으면 페이지가 자동으로 작동
- **세탁기 증감 자유** — QR 스티커 출력/부착/제거만으로 완결
- **모바일 퍼스트** — QR 스캔 → 모바일 브라우저가 주 사용 환경, PWA 확장 고려


## 9. 핵심 로직 설명

### 페이지가 자동으로 작동하는 원리 (관리자 불필요)

이 서비스에는 **관리자 계정, 관리자 페이지, 사전 등록 과정이 전혀 없다.**

핵심은 React Router의 **동적 라우팅**(`/q/:qrId`)이다. URL에 어떤 QR ID가 들어오든 같은 컴포넌트가 렌더링된다.

```
/q/A1  → QrPage 컴포넌트가 qrId="A1" 로 실행
/q/A2  → QrPage 컴포넌트가 qrId="A2" 로 실행
/q/Z99 → QrPage 컴포넌트가 qrId="Z99" 로 실행
/q/아무거나 → QrPage 컴포넌트가 qrId="아무거나" 로 실행
```

컴포넌트가 마운트되면 `useParams()`로 `qrId`를 꺼내서 Supabase에 조회할 뿐이다.

- 레코드 있음 → "사용 중" + 닉네임 + 바구니 사진 표시
- 레코드 없음 → "비어있음" + 등록 버튼 표시

**DB에 미리 세탁기를 등록하는 과정이 없다.** 누군가 QR을 스캔해서 세탁물을 등록하는 순간 비로소 DB에 레코드가 생기고, 회수 완료하면 레코드가 삭제된다.

### 세탁기 추가/삭제 방법

| 상황               | 방법                                                  | DB 작업 |
| ------------------ | ----------------------------------------------------- | ------- |
| 세탁기 추가        | QR 생성 사이트에서 URL 입력 → QR 출력 → 세탁기에 부착 | 없음    |
| 세탁기 제거        | QR 스티커를 떼면 끝                                   | 없음    |
| 세탁기 10대 → 20대 | QR 10개 더 출력해서 붙이기                            | 없음    |
| QR ID 바꾸고 싶음  | 새 URL로 QR 출력 → 기존 스티커 교체                   | 없음    |

**QR 생성 예시:**

어떤 무료 QR 사이트든 상관없다. URL만 정확히 입력하면 된다.

```
세탁기 1호 → https://bbalraeq.vercel.app/q/W1
세탁기 2호 → https://bbalraeq.vercel.app/q/W2
건조기 1호 → https://bbalraeq.vercel.app/q/D1
건조기 2호 → https://bbalraeq.vercel.app/q/D2
```

QR ID 네이밍은 자유다. `W1`, `washer-01`, `3층세탁기`, `abc123` 뭐든 된다. URL로 유효한 문자열이면 전부 작동한다.

### 세탁물 등록 흐름

1. 닉네임 입력 (클라이언트에서 2~10자 검증)
2. 바구니 사진 촬영/선택 → 이미지 압축 → Supabase Storage 업로드
3. 해당 `qr_id`로 기존 레코드가 있으면 → 기존 사진 Storage에서 삭제 + 레코드 삭제
4. 새 레코드 INSERT (`qr_id`, `nickname`, `image_url`, `created_at`)

### 사용 완료 흐름

1. 회수 완료 버튼 클릭
2. 해당 `qr_id` 레코드 삭제 + Storage 사진 삭제
3. 페이지가 "비어있음" 상태로 전환

### 사진 자동 교체 (Storage 최적화)

- 하나의 QR ID에는 항상 최대 1장의 사진만 존재
- 새 등록 시: 기존 사진 삭제 → 새 사진 업로드
- 사용 완료 시: 사진 삭제
- Storage 경로: `baskets/{qr_id}/{filename}`

## 10. 기술 스택

| 영역         | 기술                                    | 이유                                                         |
| ------------ | --------------------------------------- | ------------------------------------------------------------ |
| 프론트엔드   | React 18 + Vite                         | SPA로 충분, 빌드 빠름, PWA 전환 쉬움                         |
| 라우팅       | React Router v6                         | `/q/:qrId` 동적 라우팅으로 모든 QR 대응                      |
| 스타일링     | Tailwind CSS                            | 빠른 모바일 퍼스트 UI 구현                                   |
| UI 컴포넌트  | shadcn/ui                               | 경량, Tailwind 기반, 커스터마이징 쉬움                       |
| 언어         | TypeScript                              | 타입 안전성                                                  |
| DB + Storage | Supabase                                | 백엔드 없이 DB/Storage/API 한번에 해결                       |
| 배포         | Vercel (또는 Netlify, Cloudflare Pages) | 정적 빌드라 어디든 배포 가능                                 |
| 이미지 압축  | browser-image-compression               | 클라이언트에서 압축 후 업로드                                |
| PWA          | vite-plugin-pwa                         | 플러그인 하나로 Service Worker + 오프라인 캐싱 + 홈화면 추가 |
| QR 생성      | 무료 QR 사이트                          | MVP는 수동 생성으로 충분                                     |

### Next.js 대신 React + Vite를 선택한 이유

| 비교 항목        | Next.js                       | React + Vite                         |
| ---------------- | ----------------------------- | ------------------------------------ |
| SSR/SEO          | 지원 (이 프로젝트에선 불필요) | 미지원 (QR 스캔 접속이라 SEO 무의미) |
| PWA 전환         | next-pwa 필요 (관리 불안정)   | vite-plugin-pwa (안정적, 설정 간편)  |
| 빌드 결과물      | 서버 런타임 포함              | 순수 정적 파일 (가볍고 빠름)         |
| 배포 자유도      | Vercel에 최적화 (종속 경향)   | 어디든 배포 가능                     |
| 복잡도           | 서버 컴포넌트 등 학습 필요    | 순수 React (간단)                    |
| 모바일 로딩 속도 | 보통                          | 번들이 작아 더 빠름                  |

### 프로젝트 초기 세팅

**1단계: 프로젝트 생성**

```bash
npm create vite@latest bbalraeq -- --template react-ts
cd bbalraeq
npm install
```

이 명령으로 자동 설치되는 패키지 (별도 설치 불필요):

| 패키지               | 용도                |
| -------------------- | ------------------- |
| react                | UI 라이브러리       |
| react-dom            | React DOM 렌더링    |
| typescript           | 타입 안전성         |
| @types/react         | React 타입 정의     |
| @types/react-dom     | React DOM 타입 정의 |
| vite                 | 빌드 도구           |
| @vitejs/plugin-react | Vite React 플러그인 |

**2단계: 프로덕션 패키지 설치 (dependencies)**

```bash
npm install react-router-dom @supabase/supabase-js browser-image-compression lucide-react clsx tailwind-merge class-variance-authority
```

| 패키지                    | 용도                                       |
| ------------------------- | ------------------------------------------ |
| react-router-dom          | `/q/:qrId` 동적 라우팅                     |
| @supabase/supabase-js     | DB 조회/삽입/삭제 + Storage 업로드/삭제    |
| browser-image-compression | 바구니 사진 클라이언트 측 압축             |
| lucide-react              | 아이콘 (카메라, 체크, 뒤로가기 등)         |
| clsx                      | 조건부 className 조합 (shadcn/ui 필요)     |
| tailwind-merge            | Tailwind 클래스 충돌 방지 (shadcn/ui 필요) |
| class-variance-authority  | 컴포넌트 변형 관리 (shadcn/ui 필요)        |

**3단계: 개발 전용 패키지 설치 (devDependencies)**

```bash
npm install -D tailwindcss @tailwindcss/vite vite-plugin-pwa
```

| 패키지            | 용도                     |
| ----------------- | ------------------------ |
| tailwindcss       | 스타일링                 |
| @tailwindcss/vite | Vite용 Tailwind 플러그인 |
| vite-plugin-pwa   | PWA 지원 (MVP 이후 적용) |

**4단계: Tailwind 연결**

`vite.config.ts`:

```ts
import { defineConfig } from "vite";
import react from "@vitejs/plugin-react";
import tailwindcss from "@tailwindcss/vite";

export default defineConfig({
  plugins: [react(), tailwindcss()],
});
```

`src/index.css` 맨 위:

```css
@import "tailwindcss";
```

**5단계: 개발 서버 실행**

### 라우터 설정 (App.tsx 구조)

```
/             → Home.tsx        서비스 안내
/q/:qrId     → QrPage.tsx      QR 스캔 후 페이지 (모든 QR ID 처리)
```

`QrPage.tsx` 하나가 모든 QR을 처리한다. QR이 100개든 1000개든 코드는 이 파일 하나뿐이다. `useParams()`로 `qrId`를 꺼내서 Supabase에 조회하는 것이 전부.

### Supabase DB — 테이블 1개

**registrations**

| 컬럼       | 타입          | 설명                       |
| ---------- | ------------- | -------------------------- |
| id         | UUID (PK)     | 자동 생성                  |
| qr_id      | TEXT (UNIQUE) | QR 고유 ID (W1, W2, D1...) |
| nickname   | TEXT          | 닉네임 (2~10자)            |
| image_url  | TEXT          | 바구니 사진 URL            |
| created_at | TIMESTAMPTZ   | 등록 시간                  |

### Supabase Storage

- 버킷: `baskets`
- 경로: `baskets/{qr_id}/{filename}`
- QR ID당 사진 최대 1장 유지
